# Indicator Model

## Overview

The indicator model module provides integrated capabilities for indicator **development, management, and usage**. It supports efficient indicator creation and management by developers, and created indicators can be used as knowledge sources for agents.

Three indicator types are supported:

- **Atomic indicator**: a basic metric derived from a business event, such as `payment amount`, and cannot be decomposed further
- **Derived indicator**: built from `atomic indicator + modifier + time period`, such as `payment amount by overseas buyers in the last 1 day`
- **Composite indicator**: generated by combining and calculating atomic, derived, or other composite indicators

## Create an Indicator Model

### Prerequisites

Before creating an indicator model, complete the following:

1. Configure data connections under **Multimodal Data Lake > VEGA Virtualization > Data Connections**
2. Configure data views under **Multimodal Data Lake > VEGA Virtualization > Data Views**

### Create an Atomic Indicator

Atomic indicators are generated from fact-table metric fields. Open **VEGA Virtualization > Data Model > Indicator Model**, click **New**, and select **Atomic Indicator**.

#### Step 1: Basic Configuration

| Configuration Item | Description |
| --- | --- |
| Indicator Name | Enter a business-meaningful name, such as `Order Payment Amount`. Required |
| ID | Optional custom unique identifier. If not provided, the system generates it automatically |
| Group | Optional business group for classification |
| Tags | Optional keyword tags such as `payment business` or `core indicator` for easier search |
| Remarks | Optional explanation of the indicator |

After completing this step, click **Next**.

#### Step 2: Model Configuration

First select the source view, which must be created in advance, and then configure it based on the view type, either VEGA or index.

##### VEGA-Type View

| Configuration Item | Description |
| --- | --- |
| Data Filter | Optional. Supports SQL filters or field-based conditions |
| Metric Calculation | Required. Choose SQL aggregate formula or field aggregation |
| Analysis Dimensions | Optional. Add or remove custom dimensions in addition to the default date-time dimension |
| Grouping Field | Optional. Choose the field used for grouping |
| Result Sorting | Optional. Set sorting priority based on grouping fields |
| Result Filtering | Optional. Add conditions for filtering the result set |
| Date-Time Identifier | Required. Select a date, datetime, or timestamp field from the view |
| Indicator Unit | Required. Choose a unit such as `CNY`, `times`, or `items` |

##### Index-Type View

| Configuration Item | Description |
| --- | --- |
| Query Language | Required. Choose `PromQL` or `DSL` |
| Formula | Required. Enter the formula according to the selected query language |
| Metric Field | Required when using DSL. If value aggregation is `top_hits`, fill in the numeric field under `_source`; otherwise, fill in the value aggregation name |
| Indicator Unit | Required. Choose a unit such as `times` or `GB` |

**DSL formula rules**:

- It must include `date_histogram` and `value_count` aggregations
- Nested aggregations are supported up to 7 levels
- Parallel aggregations are not supported
- The preset DSL structure includes a filter query with `size=0` and a three-level nested aggregation
- Custom DSL must follow OpenSearch aggregation syntax

#### Step 3: Persistence Configuration (Optional)

Enable the **Persistence** switch to store indicator data in a specified index library. Configure:

- Index library
- Retention period
- Update frequency

#### Step 4: Finish Creation

Click **Confirm** to return to the indicator list. You can click **Data Preview** in the actions column of the target indicator to inspect trends with filters applied.

### Create a Derived Indicator

Derived indicators are based on atomic indicators. Open **VEGA Virtualization > Data Model > Indicator Model**, click **New**, and choose **Derived Indicator**.

#### Step 1: Basic Configuration

Same as for atomic indicators.

#### Step 2: Data Source Configuration

- Select the base atomic indicator from the existing list
- Data filtering, result sorting, and result filtering are the same as for atomic indicators
- Analysis dimensions inherit from the atomic indicator and can be adjusted
- Indicator units follow the same rule as for atomic indicators

#### Step 3: Finish Creation

Click **Confirm**. Use **Data Preview** to inspect the result.

### Create a Composite Indicator

Composite indicators are based on atomic, derived, or other composite indicators. Open **VEGA Virtualization > Data Model > Indicator Model**, click **New**, and choose **Composite Indicator**.

#### Step 1: Basic Configuration

Same as for atomic indicators.

#### Step 2: Model Configuration

- Add referenced indicators from the left-side list into the expression editor
- Edit the calculation expression using operators, such as `Indicator A - Indicator B`
- Data filtering, result sorting, result filtering, and analysis dimensions follow the same rules as atomic indicators
- Choose an indicator unit according to the calculation logic, such as `CNY` or `percentage`

#### Step 3: Finish Creation

Click **Confirm**. Use **Data Preview** to inspect the result.

### Import and Export Indicator Models

- **Import**: click **Import** and select a JSON indicator configuration file that follows the system template
- **Export**: select target indicators in the list, click **Export**, and generate a JSON file

## Notes

1. **Indicator dependencies**
   - Derived indicators must be based on atomic indicators and cannot be built directly on other derived indicators
   - Composite indicators cannot extend beyond the dimension scope of their base atomic indicators and cannot add new dimensions
2. **DSL syntax validation**
   - The system validates PromQL and DSL syntax automatically before saving
   - Field permissions are not validated currently. Querying a non-existent field returns empty results
3. **Data preview**
   - Choose a time range and dimension filters before previewing
   - If the dataset is large, preview may be delayed, so narrowing the time range is recommended
