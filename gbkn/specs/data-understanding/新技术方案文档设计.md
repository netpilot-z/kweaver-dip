# 库表数据理解需求方案设计

> 基于业务对象管理界面的数据表结构设计（`t_data_view` 体系）
> 创建时间: 2026-01-30
> 更新时间: 2026-04-10（迁移为 `t_data_view` + 临时表/正式表分层）

## 一、设计概述

本设计方案实现了业务语义与数据存储的解耦，核心思想是：

- **业务对象（Business Object）**：业务实体的抽象
- **属性（Attribute）**：业务语义的抽象，关联数据视图字段语义补全结果
- **数据分层**：理解阶段统一写入临时表，提交阶段统一 Upsert 到正式表

## 三、数据表设计

> **【强制执行】本章节定义的所有数据表结构必须严格按照当前设计执行，不得擅自修改字段名称、类型、约束或索引。任何修改需经过完整评审流程。**

### 3.1 业务对象表 `t_business_object`

存储业务对象的基本信息，区分已发布业务对象和候选业务对象。

```sql
CREATE TABLE IF NOT EXISTS t_business_object (
    id             CHAR(36)     NOT NULL                       COMMENT '业务对象UUID（主键）',
    object_name    VARCHAR(100) NOT NULL                       COMMENT '业务对象名称',
    object_type    TINYINT      NOT NULL DEFAULT 0             COMMENT '对象类型：0-候选业务对象,1-已发布业务对象',
    form_view_id   CHAR(36)     NOT NULL                       COMMENT '关联数据视图UUID',
    status         TINYINT      NOT NULL DEFAULT 1             COMMENT '状态：0-禁用,1-启用',
    created_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at     DATETIME(3)          DEFAULT NULL           COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务对象表';
```

### 3.2 业务对象属性表 `t_business_object_attributes`

存储业务对象的属性定义，关联到字段语义补全正式表 `t_data_view_field_semantic` 获取字段语义信息。

```sql
CREATE TABLE IF NOT EXISTS t_business_object_attributes (
    id                   CHAR(36)     NOT NULL                       COMMENT '属性UUID（主键）',

    attr_name            VARCHAR(100) NOT NULL                       COMMENT '属性名称',
    created_at           DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at           DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at           DATETIME(3)          DEFAULT NULL           COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务对象属性表';
```

### 3.4 数据视图理解结果表 `t_data_view_understanding`

用于保存每个数据视图的理解结果（理解状态、视图业务名称、视图描述），并冗余保存视图技术信息（来源 `t_data_view`）。该表按 `view_id` 唯一，重复理解使用更新策略覆盖最新结果。

```sql
CREATE TABLE IF NOT EXISTS t_data_view_understanding (
    id                  CHAR(36)     NOT NULL COMMENT '记录UUID（主键）',
    view_id             VARCHAR(40)  NOT NULL COMMENT '关联 t_data_view.f_view_id',
    view_technical_name VARCHAR(255) NOT NULL DEFAULT '' COMMENT '视图技术名称（冗余，来源 t_data_view.f_technical_name）',
    view_group_id       VARCHAR(40)  NOT NULL DEFAULT '' COMMENT '视图分组ID（冗余，来源 t_data_view.f_group_id）',
    view_data_source_id VARCHAR(40)  NOT NULL DEFAULT '' COMMENT '数据源ID（冗余，来源 t_data_view.f_data_source_id）',
    view_type           VARCHAR(10)  NOT NULL DEFAULT '' COMMENT '视图类型（冗余，来源 t_data_view.f_type）',
    view_comment        VARCHAR(255) NOT NULL DEFAULT '' COMMENT '备注（冗余，来源 t_data_view.f_comment）',
    understand_status   TINYINT      NOT NULL DEFAULT 0 COMMENT '理解状态：0-未理解,1-理解中,2-待确认,3-已完成,4-已发布,5-理解失败',
    table_business_name VARCHAR(255) DEFAULT NULL COMMENT '视图业务名称',
    table_description   VARCHAR(300) DEFAULT NULL COMMENT '视图描述',
    created_at          DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at          DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at          DATETIME(3)  DEFAULT NULL COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (id),
    UNIQUE KEY uk_view_id (view_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='数据视图理解结果表';
```

**冗余字段来源规则（来自 `t_data_view`）**：

| `t_data_view_understanding` 字段 | 来源字段 |
| --- | --- |
| `view_technical_name` | `t_data_view.f_technical_name` |
| `view_group_id` | `t_data_view.f_group_id` |
| `view_data_source_id` | `t_data_view.f_data_source_id` |
| `view_type` | `t_data_view.f_type` |
| `view_comment` | `t_data_view.f_comment` |

**更新策略**：

- 触发理解时，先按 `view_id` 读取 `t_data_view` 最新数据并写入（或更新）冗余字段。
- 提交时执行 Upsert：若 `view_id` 已存在则更新冗余字段与理解结果，不存在则新增。

**理解状态说明：**

| 状态值 | 状态名称 | 说明                   |
| ------ | -------- | ---------------------- |
| 0      | 未理解   | 尚未进行库表理解操作     |
| 1      | 理解中   | 正在进行库表理解操作     |
| 2      | 待确认   | 理解完成，待用户确认     |
| 3      | 已完成   | 已完成库表理解          |
| 4      | 已发布   | 业务对象已正式发布      |
| 5      | 理解失败 | AI理解失败，可重新尝试 |

### 3.5 字段语义补全正式表 `t_data_view_field_semantic`

用于保存提交确认后的字段语义结果，和临时表 `t_form_view_field_info_temp` 对应。

> **【重要】本表包含从 t_data_view.f_fields JSON 解析后的冗余字段，查询时不再解析 JSON，提升性能。**

**JSON字段来源（t_data_view.f_fields）：**
```json
{
  "name": "id",                    // 字段技术名称
  "type": "string",                // 字段类型
  "comment": "",                   // 字段注释
  "display_name": "id",            // 显示名称（可作为业务名称）
  "original_name": "id",           // 原始字段名称
  "data_length": 255,              // 数据长度
  "data_accuracy": 0,              // 数据精度
  "status": "modify",              // 状态
  "is_nullable": "YES",            // 是否可空
  "business_timestamp": false,      // 是否业务时间戳
  "is_primary_key": { ... }       // 是否主键
}
```

```sql
CREATE TABLE IF NOT EXISTS t_data_view_field_semantic (
    id                      CHAR(36)     NOT NULL COMMENT '记录UUID（主键）',
    view_id                 VARCHAR(40)  NOT NULL COMMENT '关联数据视图ID（对应 t_data_view.f_view_id）',
    view_field_id           VARCHAR(64)  NOT NULL COMMENT '关联字段ID（对应 JSON 中的 name 字段）',

    -- 字段基础信息（冗余，从 t_data_view.f_fields JSON 解析，避免重复解析）
    field_tech_name         VARCHAR(255) DEFAULT NULL COMMENT '字段技术名称（对应 JSON.name）',
    field_type              VARCHAR(64)  DEFAULT NULL COMMENT '字段类型（对应 JSON.type）',
    field_comment          VARCHAR(500) DEFAULT NULL COMMENT '字段注释（对应 JSON.comment）',
    field_display_name     VARCHAR(255) DEFAULT NULL COMMENT '字段显示名称（对应 JSON.display_name）',
    field_original_name    VARCHAR(255) DEFAULT NULL COMMENT '原始字段名称（对应 JSON.original_name）',
    field_data_length      INT          DEFAULT NULL COMMENT '数据长度（对应 JSON.data_length）',
    field_data_accuracy    INT          DEFAULT NULL COMMENT '数据精度（对应 JSON.data_accuracy）',
    field_is_nullable      VARCHAR(10)  DEFAULT NULL COMMENT '是否可空（对应 JSON.is_nullable）',
    field_is_primary_key   TINYINT      DEFAULT NULL COMMENT '是否主键（对应 JSON.is_primary_key.Bool，1-是，0-否）',
    field_business_timestamp TINYINT    DEFAULT NULL COMMENT '是否业务时间戳（对应 JSON.business_timestamp）',

    -- 字段语义补全（AI 分析结果，提交后落库）
    field_business_name     VARCHAR(255) DEFAULT NULL COMMENT '字段业务名称（AI 分析）',
    field_role              TINYINT      DEFAULT NULL COMMENT '字段角色：1-业务主键, 2-关联标识, 3-业务状态, 4-时间字段, 5-业务指标, 6-业务特征, 7-审计字段, 8-技术字段',
    field_description       VARCHAR(300) DEFAULT NULL COMMENT '字段描述（AI 分析）',

    created_at              DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at              DATETIME(3)  DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at              DATETIME(3)  DEFAULT NULL COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (id),
    UNIQUE KEY uk_view_field (view_id, view_field_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='字段语义补全正式表';
```

### 3.6 业务对象临时表 `t_business_object_temp`

用于在用户提交确认前临时保存业务对象的编辑操作，支持版本控制。

```sql
CREATE TABLE IF NOT EXISTS t_business_object_temp (
    id             CHAR(36)     NOT NULL                       COMMENT '业务对象UUID（主键）',
    form_view_id   CHAR(36)     NOT NULL                       COMMENT '关联数据视图UUID',
    in_use        TINYINT      NOT NULL DEFAULT 0            COMMENT '当前使用标识：0=历史版本, 1=当前使用',
    user_id        CHAR(36)                                         COMMENT '为空代表模型操作，不为空代表某用户操作',
    version        INT          NOT NULL DEFAULT 10            COMMENT '版本号（存储格式：10=1.0，11=1.1，每次递增1表示0.1版本）',
    object_name    VARCHAR(100) NOT NULL                       COMMENT '业务对象名称',
    created_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    updated_at     DATETIME(3)          DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    deleted_at     DATETIME(3)          DEFAULT NULL           COMMENT '删除时间(逻辑删除)',
    PRIMARY KEY (id),
    KEY idx_form_view_version (form_view_id, version, deleted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='业务对象临时表';
```

**数据来源：**

1. **AI 自动生成**：通过订阅 Kafka 消息 `data-understanding-responses`，接收 AI 服务分析完成后返回的业务对象数据
2. **用户手动编辑**：用户在页面上编辑业务对象名称时保存的数据

**使用说明：**

- 用户在页面编辑业务对象时，先保存到此临时表
- 点击提交确认后，将数据迁移到正式表 `t_business_object`
- `user_id` 为空时代表 AI 模型自动操作（通过 Kafka 订阅），不为空时代表具体用户操作
- `in_use` 用于标识当前使用的版本：`1=当前使用`，`0=历史版本`
  - AI 每次生成新识别结果时，会自动将新版本的 `in_use` 设置为 1，历史版本设置为 0
  - 删除识别结果时，只删除 `in_use = 1` 的数据
- AI 自动生成时递增版本号，初始版本为10（显示为1.0），用户手动编辑时版本号不变
- 支持重新识别，AI 重新识别时会创建新版本记录，旧版本记录保留作为历史

**使用说明：**

### 3.8 库表信息临时表 `t_form_view_info_temp`

```sql
    form_view_id         CHAR(36)     NOT NULL                       COMMENT '关联数据视图UUID',

    version              INT          NOT NULL DEFAULT 10            COMMENT '版本号（存储格式：10=1.0，11=1.1，每次递增1表示0.1版本）',

    -- 库表级别语义补全
    table_business_name  VARCHAR(255)        DEFAULT NULL            COMMENT '库表业务名称',
    table_description    VARCHAR(300)        DEFAULT NULL            COMMENT '库表描述',

    deleted_at           DATETIME(3)          DEFAULT NULL           COMMENT '删除时间(逻辑删除)',

    PRIMARY KEY (id),
    KEY idx_form_view_version (form_view_id, version, deleted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='库表信息临时表';
```

### 4.3 接口详细说明

#### 4.3.1 查询字段语义补全数据

**接口路径**：`GET /api/data-semantic/v1/{id}/fields`

**请求参数**：

| 参数名          | 类型    | 必填 | 说明                                                                                               |
| --------------- | ------- | ---- | -------------------------------------------------------------------------------------------------- |
| id              | string  | 是   | 数据视图UUID                                                                                       |
| keyword         | string  | 否   | 搜索字段技术名称（模糊匹配 field_tech_name）                                                       |
| only_incomplete | boolean | 否   | 是否仅查看未完成理解的字段（字段业务名称、字段角色、字段描述任意一个为空即视为未完成），默认 false |

**处理逻辑**：

1. **查询当前理解状态**

   ```sql
   SELECT understand_status
   FROM t_data_view_understanding
   WHERE view_id = {view_id} AND deleted_at IS NULL;
   ```
2. **状态判断处理**

   **2.1 状态为 1-理解中**

   返回错误响应：

   ```json
   {
     "code": 400,
     "message": "当前正在理解中，请稍后再试",
     "data": {
       "error_type": "understanding_in_progress",
       "understand_status": 1
     }
   }
   ```

   **2.2 状态为 0-未理解、3-已完成、4-已发布**

   从正式表查询数据：

   ```sql
   -- 查询视图基础信息与理解结果
   SELECT
     dv.f_technical_name AS table_tech_name,
     dvu.table_business_name,
     dvu.table_description
   FROM t_data_view dv
   LEFT JOIN t_data_view_understanding dvu ON dv.f_view_id = dvu.view_id AND dvu.deleted_at IS NULL
   WHERE dv.f_view_id = {view_id} AND dv.f_delete_time = 0;

   -- 查询字段信息（字段基础信息来自冗余字段，语义信息来自正式表）
   SELECT
     fsem.view_field_id AS form_view_field_id,
     fsem.field_tech_name,
     fsem.field_type,
     fsem.field_business_name,
     fsem.field_role,
     fsem.field_description
   FROM t_data_view_field_semantic fsem
   WHERE fsem.view_id = {view_id}
     AND fsem.deleted_at IS NULL
   ORDER BY fsem.id ASC;
   ```

   **2.3 状态为 2-待确认**

   从临时表获取字段属性信息，字段基础信息从临时表冗余字段获取：

   ```sql
   -- 获取当前版本号
   SELECT COALESCE(MAX(version), 10) AS current_version
   FROM t_form_view_info_temp
   WHERE form_view_id = {view_id} AND deleted_at IS NULL;

   -- 查询库表信息（临时表优先，基础信息来自 t_data_view）
   SELECT
     dv.f_technical_name AS table_tech_name,
     COALESCE(fvit.table_business_name, dvu.table_business_name) AS table_business_name,
     COALESCE(fvit.table_description, dvu.table_description) AS table_description
   FROM t_data_view dv
   LEFT JOIN t_data_view_understanding dvu
     ON dv.f_view_id = dvu.view_id AND dvu.deleted_at IS NULL
   LEFT JOIN t_form_view_info_temp fvit
     ON dv.f_view_id = fvit.form_view_id
     AND fvit.version = {current_version}
     AND fvit.deleted_at IS NULL
   WHERE dv.f_view_id = {view_id} AND dv.f_delete_time = 0;

   -- 查询字段信息（字段属性从临时表冗余字段，语义信息临时表优先）
   -- 先查临时表有而正式表没有的，再查临时表和正式表都有的
   SELECT
     fvfit.form_view_field_id AS form_view_field_id,
     fvfit.field_tech_name,
     fvfit.field_type,
     COALESCE(fvfit.field_business_name, fsem.field_business_name) AS field_business_name,
     COALESCE(fvfit.field_role, fsem.field_role) AS field_role,
     COALESCE(fvfit.field_description, fsem.field_description) AS field_description
   FROM t_form_view_field_info_temp fvfit
   LEFT JOIN t_data_view_field_semantic fsem
     ON fvfit.form_view_field_id = fsem.view_field_id
    AND fvfit.form_view_id = fsem.view_id
    AND fsem.deleted_at IS NULL
   WHERE fvfit.form_view_id = {view_id}
     AND fvfit.version = {current_version}
     AND fvfit.deleted_at IS NULL
   UNION
   SELECT
     fsem.view_field_id AS form_view_field_id,
     fsem.field_tech_name,
     fsem.field_type,
     fsem.field_business_name,
     fsem.field_role,
     fsem.field_description
   FROM t_data_view_field_semantic fsem
   WHERE fsem.view_id = {view_id}
     AND fsem.deleted_at IS NULL
     AND NOT EXISTS (
       SELECT 1 FROM t_form_view_field_info_temp fvfit
       WHERE fvfit.form_view_field_id = fsem.view_field_id
         AND fvfit.form_view_id = fsem.view_id
         AND fvfit.version = {current_version}
         AND fvfit.deleted_at IS NULL
     )
   ORDER BY form_view_field_id ASC;
   ```
3. **参数过滤处理**

   **3.1 keyword 不为空时，模糊匹配字段技术名称**

   ```sql
   -- 在 WHERE 子句中添加条件
   AND fsem.field_tech_name LIKE CONCAT('%', {keyword}, '%')
   ```

   **3.2 only_incomplete = true 时，过滤未完成的字段**

   *状态为 0-未理解、3-已完成、4-已发布时（查询正式表）：*

   ```sql
   -- 在 WHERE 子句中添加条件
   AND (
     business_name IS NULL OR business_name = ''
     OR field_role IS NULL
     OR field_description IS NULL OR field_description = ''
   )
   ```

   *状态为 2-待确认时（查询临时表）：*

   ```sql
   -- 在 WHERE 子句中添加条件
   AND (
     fvfit.field_business_name IS NULL OR fvfit.field_business_name = ''
     OR fvfit.field_role IS NULL
     OR fvfit.field_description IS NULL OR fvfit.field_description = ''
   )
   ```

**数据来源说明**：

根据 `understand_status` 状态决定数据来源：

| 状态     | 库表信息                            | 字段信息                         | 字段基本信息          |
| -------- | -------------- | --------------- | --------------- |
| 0-未理解 | t_data_view + t_data_view_understanding | t_data_view_field_semantic | 正式表冗余字段 |
| 1-理解中 | 禁止调用       | -               | -               |
| 2-待确认 | 临时表（优先） | 临时表（优先）  | 临时表冗余字段 |
| 3-已完成 | t_data_view + t_data_view_understanding | t_data_view_field_semantic | 正式表冗余字段 |
| 4-已发布 | t_data_view + t_data_view_understanding | t_data_view_field_semantic | 正式表冗余字段 |

- `table_tech_name`：来自 `t_data_view.f_technical_name`
- `table_business_name`、`table_description`：状态0/3/4来自 `t_data_view_understanding`，状态2临时表优先
- `field_tech_name`、`field_type`：状态0/3/4来自 `t_data_view_field_semantic` 冗余字段，状态2来自 `t_form_view_field_info_temp` 冗余字段
- `field_business_name`、`field_role`、`field_description`：状态0/3/4来自正式表，状态2临时表优先
- `current_version`：状态2时返回当前版本号，其他状态返回 0

**响应数据**：

```json
{
  "code": 200,                                 // int - 响应状态码
  "data": {
    "current_version": 10,                     // int - 当前版本号（存储格式：10=1.0，显示时除以10）
    "table_business_name": "员工信息表",       // string - 库表业务名称
    "table_tech_name": "cowenrr",            // string - 库表技术名称（来自 t_data_view.f_technical_name）
    "table_description": "员工基础信息表",    // string - 库表描述
    "fields": [                               // array - 字段级别语义补全信息列表
      {
        "form_view_field_id": "xxx",           // string - 字段UUID
        "field_business_name": "员工姓名",      // string - 字段业务名称
        "field_tech_name": "name",             // string - 字段技术名称（来自冗余字段）
        "field_type": "VARCHAR",               // string - 字段类型（来自冗余字段）
        "field_role": 1,                       // int - 字段角色（1-业务主键, 2-关联标识, 3-业务状态, 4-时间字段, 5-业务指标, 6-业务特征, 7-审计字段, 8-技术字段）
        "field_description": "员工真实姓名"    // string - 字段描述
      }
    ]
  }
}
```

**说明**：

- `current_version` 通过 `COALESCE(MAX(version), 10)` 获取最大版本号（存储格式：10=1.0，显示时除以10）
- 返回的数据为最新版本的数据（通过 `in_use = 1` 筛选当前使用的版本）

#### 4.3.2 查询业务对象识别结果

**接口路径**：`GET /api/data-semantic/v1/{id}/business-objects`

| 参数名    | 类型   | 必填 | 说明                                                                |
| --------- | ------ | ---- | ------------------------------------------------------------------- |
| id        | string | 是   | 数据视图UUID                                                        |
| object_id | int    | 否   | 业务对象ID（传此参数则返回单个业务对象的详细数据）                  |
| keyword   | string | 否   | 搜索属性名称/字段业务名称（模糊匹配 attr_name/field_business_name） |

**处理逻辑**：

1. **查询当前理解状态**

   ```sql
   SELECT understand_status
   FROM t_data_view_understanding
   WHERE view_id = {view_id} AND deleted_at IS NULL;
   ```
2. **状态判断处理**

   **2.1 状态为 1-理解中**

   返回错误响应：

   ```json
   {
     "code": 400,
     "message": "当前正在理解中，请稍后再试",
     "data": {
       "error_type": "understanding_in_progress",
       "understand_status": 1
     }
   }
   ```

   **2.2 状态为 0-未理解、3-已完成、4-已发布**

   从正式表查询数据：

   ```sql
   -- 查询业务对象列表
   SELECT
     bo.id AS object_id,
     bo.uuid AS object_uuid,
     bo.object_name
   FROM t_business_object bo
   WHERE bo.form_view_id = {view_id}
     AND bo.deleted_at IS NULL
   ORDER BY bo.id ASC;

   -- 查询业务对象属性（字段基础信息来自冗余字段，语义信息来自正式表）
   SELECT
     boa.id AS attribute_id,
     boa.business_object_id,
     boa.attr_name,
     boa.form_view_field_id,
     fsem.field_tech_name,
     fsem.field_type,
     fsem.field_business_name,
     fsem.field_role
   FROM t_business_object_attributes boa
   LEFT JOIN t_data_view_field_semantic fsem
     ON fsem.view_id = boa.form_view_id
    AND fsem.view_field_id = boa.form_view_field_id
    AND fsem.deleted_at IS NULL
   WHERE boa.form_view_id = {view_id}
     AND boa.deleted_at IS NULL
   ORDER BY boa.id ASC;
   ```

   **2.3 状态为 2-待确认**

   从临时表获取业务对象和属性信息，字段基础信息从临时表冗余字段获取：

   ```sql
   -- 获取当前版本号
   SELECT COALESCE(MAX(version), 10) AS current_version
   FROM t_business_object_temp
   WHERE form_view_id = {view_id} AND deleted_at IS NULL;

   -- 查询业务对象列表
   SELECT
     bot.id AS object_id,
     bot.uuid AS object_uuid,
     bot.object_name
   FROM t_business_object_temp bot
   WHERE bot.form_view_id = {view_id}
     AND bot.version = {current_version}
     AND bot.deleted_at IS NULL
   ORDER BY bot.id ASC;

   -- 查询业务对象属性（字段基础信息来自临时表冗余字段，语义信息临时表优先）
   SELECT
     boattr.id AS attribute_id,
     boattr.business_object_id,
     boattr.attr_name,
     boattr.form_view_field_id,
     COALESCE(fvfit.field_tech_name, fsem.field_tech_name) AS field_tech_name,
     COALESCE(fvfit.field_type, fsem.field_type) AS field_type,
     COALESCE(fvfit.field_business_name, fsem.field_business_name) AS field_business_name,
     COALESCE(fvfit.field_role, fsem.field_role) AS field_role
   FROM t_business_object_attributes_temp boattr
   LEFT JOIN t_form_view_field_info_temp fvfit
     ON boattr.form_view_field_id = fvfit.form_view_field_id
    AND boattr.form_view_id = fvfit.form_view_id
    AND fvfit.version = {current_version}
    AND fvfit.deleted_at IS NULL
   LEFT JOIN t_data_view_field_semantic fsem
     ON boattr.form_view_field_id = fsem.view_field_id
    AND boattr.form_view_id = fsem.view_id
    AND fsem.deleted_at IS NULL
   WHERE boattr.form_view_id = {view_id}
     AND boattr.version = {current_version}
     AND boattr.deleted_at IS NULL
   ORDER BY boattr.id ASC;
   ```
3. **参数过滤处理**

   **3.1 object_id 不为空时，只返回指定业务对象**

   ```sql
   -- 添加 WHERE 条件
   AND bo.id = {object_id}  -- 正式表
   -- 或
   AND bot.id = {object_id} -- 临时表
   ```

   **3.2 keyword 不为空时，模糊匹配属性名称/字段业务名称**

   *状态为 0-未理解、3-已完成、4-已发布时（查询正式表）：*

   ```sql
   -- 在 WHERE 子句中添加条件
   AND (
     boa.attr_name LIKE CONCAT('%', {keyword}, '%')
    OR fsem.field_business_name LIKE CONCAT('%', {keyword}, '%')
   )
   ```

   *状态为 2-待确认时（查询临时表）：*

   ```sql
   -- 在 WHERE 子句中添加条件
   AND (
     boattr.attr_name LIKE CONCAT('%', {keyword}, '%')
    OR COALESCE(fvfit.field_business_name, fsem.field_business_name) LIKE CONCAT('%', {keyword}, '%')
   )
   ```
4. **组装返回数据**

   将业务对象和属性数据组装成嵌套结构：

   - 对每个业务对象，查找其关联的属性列表

#### 4.3.3 提交理解数据

**接口路径**：`POST /api/data-semantic/v1/{id}/submit`

**请求参数**：

| 参数名 | 类型   | 必填 | 说明         |
| ------ | ------ | ---- | ------------ |
| id     | string | 是   | 数据视图UUID |

**并发控制**：同一用户对同一库表1秒内只允许点击一次，使用 Redis 限流，Key 格式为 `库表UUID:用户ID`。

**处理逻辑**：

1. **校验当前 understand_status**

   ```sql
   SELECT understand_status FROM t_data_view_understanding WHERE view_id = {view_id} AND deleted_at IS NULL;
   ```

   - `0-未理解`：数据尚未生成，不允许提交
   - `1-理解中`：正在生成数据，不允许提交
   - `2-待确认`：允许提交
   - `3-已完成`：允许提交（同步临时表数据到正式表）
   - `4-已发布`：已发布，不允许重新提交
2. **检查临时表数据并同步**

   2.1 **检查库表信息临时表是否有数据**

   ```sql
   SELECT COUNT(*) AS count
   FROM t_form_view_info_temp
   WHERE form_view_id = {form_view_id} AND deleted_at IS NULL;
   ```

   - 如果有数据：获取当前版本号并同步到正式表
   - 如果无数据：跳过库表信息同步

   2.2 **检查字段信息临时表是否有数据**

   ```sql
   SELECT COUNT(*) AS count
   FROM t_form_view_field_info_temp
   WHERE form_view_id = {form_view_id} AND deleted_at IS NULL;
   ```

   - 如果有数据：获取当前版本号并同步到正式表
   - 如果无数据：跳过字段信息同步
3. **使用数据库事务执行数据同步**

   **事务开始：**

   3.1 **同步库表信息到正式表（如果临时表有数据）**

   ```sql
   INSERT INTO t_data_view_understanding (
     id, view_id, view_technical_name, view_group_id, view_data_source_id, view_type, view_comment,
     understand_status, table_business_name, table_description, created_at, updated_at
   )
   SELECT
     UUID(), dv.f_view_id, dv.f_technical_name, dv.f_group_id, dv.f_data_source_id, dv.f_type, dv.f_comment,
     3,
     fvit.table_business_name,
     fvit.table_description,
     NOW(3), NOW(3)
   FROM t_data_view dv
   JOIN t_form_view_info_temp fvit ON dv.f_view_id = fvit.form_view_id
   WHERE dv.f_view_id = {view_id}
     AND dv.f_delete_time = 0
     AND fvit.version = {current_version}
     AND fvit.deleted_at IS NULL
   ON DUPLICATE KEY UPDATE
     view_technical_name = VALUES(view_technical_name),
     view_group_id = VALUES(view_group_id),
     view_data_source_id = VALUES(view_data_source_id),
     view_type = VALUES(view_type),
     view_comment = VALUES(view_comment),
     table_business_name = COALESCE(VALUES(table_business_name), t_data_view_understanding.table_business_name),
     table_description = COALESCE(VALUES(table_description), t_data_view_understanding.table_description),
     understand_status = VALUES(understand_status),
     updated_at = NOW(3);
   ```

   3.2 **同步字段信息到正式表（如果临时表有数据）**

   ```sql
   INSERT INTO t_data_view_field_semantic (
     id, view_id, view_field_id,
     field_tech_name, field_type,
     field_comment, field_display_name, field_original_name,
     field_data_length, field_data_accuracy, field_is_nullable, field_is_primary_key, field_business_timestamp,
     field_business_name, field_role, field_description,
     created_at, updated_at
   )
   SELECT
     UUID(), fvfit.form_view_id, fvfit.form_view_field_id,
     fvfit.field_tech_name, fvfit.field_type,
     fvfit.field_comment, fvfit.field_display_name, fvfit.field_original_name,
     fvfit.field_data_length, fvfit.field_data_accuracy, fvfit.field_is_nullable, fvfit.field_is_primary_key, fvfit.field_business_timestamp,
     fvfit.field_business_name, fvfit.field_role, fvfit.field_description,
     NOW(3), NOW(3)
   FROM t_form_view_field_info_temp fvfit
   WHERE fvfit.form_view_id = {view_id}
     AND fvfit.version = {current_version}
     AND fvfit.deleted_at IS NULL
   ON DUPLICATE KEY UPDATE
     field_tech_name = COALESCE(VALUES(field_tech_name), t_data_view_field_semantic.field_tech_name),
     field_type = COALESCE(VALUES(field_type), t_data_view_field_semantic.field_type),
     field_comment = COALESCE(VALUES(field_comment), t_data_view_field_semantic.field_comment),
     field_display_name = COALESCE(VALUES(field_display_name), t_data_view_field_semantic.field_display_name),
     field_original_name = COALESCE(VALUES(field_original_name), t_data_view_field_semantic.field_original_name),
     field_data_length = COALESCE(VALUES(field_data_length), t_data_view_field_semantic.field_data_length),
     field_data_accuracy = COALESCE(VALUES(field_data_accuracy), t_data_view_field_semantic.field_data_accuracy),
     field_is_nullable = COALESCE(VALUES(field_is_nullable), t_data_view_field_semantic.field_is_nullable),
     field_is_primary_key = COALESCE(VALUES(field_is_primary_key), t_data_view_field_semantic.field_is_primary_key),
     field_business_timestamp = COALESCE(VALUES(field_business_timestamp), t_data_view_field_semantic.field_business_timestamp),
     field_business_name = COALESCE(VALUES(field_business_name), t_data_view_field_semantic.field_business_name),
     field_role = COALESCE(VALUES(field_role), t_data_view_field_semantic.field_role),
     field_description = COALESCE(VALUES(field_description), t_data_view_field_semantic.field_description),
     updated_at = NOW(3);
   ```

  3.3 **增量同步业务对象数据到正式表**

  ```sql
  -- 按业务唯一键（form_view_id, object_name）执行 Upsert
  INSERT INTO t_business_object (
    id, object_name, object_type, form_view_id, status, created_at, updated_at, deleted_at
  )
  SELECT
    bot.id, bot.object_name, 1 AS object_type, bot.form_view_id, 1 AS status,
    NOW(3), NOW(3), NULL
  FROM t_business_object_temp bot
  WHERE bot.form_view_id = {form_view_id}
    AND bot.version = {current_version}
    AND bot.deleted_at IS NULL
  ON DUPLICATE KEY UPDATE
    object_name = VALUES(object_name),
    status = VALUES(status),
    deleted_at = NULL,
    updated_at = NOW(3);
  ```

  ```sql
  -- 逻辑删除正式表中已不存在于当前临时版本的业务对象
  UPDATE t_business_object bo
  SET bo.deleted_at = NOW(3),
      bo.updated_at = NOW(3)
  WHERE bo.form_view_id = {form_view_id}
    AND bo.deleted_at IS NULL
    AND NOT EXISTS (
      SELECT 1
      FROM t_business_object_temp bot
      WHERE bot.form_view_id = bo.form_view_id
        AND bot.version = {current_version}
        AND bot.deleted_at IS NULL
        AND bot.object_name = bo.object_name
    );
  ```

  3.4 **增量同步业务对象属性数据到正式表**

  ```sql
  -- 先按业务对象名称将属性临时记录映射到正式业务对象，再执行 Upsert
  INSERT INTO t_business_object_attributes (
    id, form_view_id, business_object_id, form_view_field_id, attr_name, created_at, updated_at, deleted_at
  )
  SELECT
    boat.id,
    boat.form_view_id,
    bo.id AS business_object_id,
    boat.form_view_field_id,
    boat.attr_name,
    NOW(3), NOW(3), NULL
  FROM t_business_object_attributes_temp boat
  JOIN t_business_object_temp bot
    ON boat.business_object_id = bot.id
   AND boat.form_view_id = bot.form_view_id
   AND boat.version = bot.version
   AND bot.deleted_at IS NULL
  JOIN t_business_object bo
    ON bo.form_view_id = bot.form_view_id
   AND bo.object_name = bot.object_name
   AND bo.deleted_at IS NULL
  WHERE boat.form_view_id = {form_view_id}
    AND boat.version = {current_version}
    AND boat.deleted_at IS NULL
  ON DUPLICATE KEY UPDATE
    attr_name = VALUES(attr_name),
    deleted_at = NULL,
    updated_at = NOW(3);
  ```

  ```sql
  -- 逻辑删除正式表中已不存在于当前临时版本的属性
  UPDATE t_business_object_attributes boa
  JOIN t_business_object bo ON bo.id = boa.business_object_id AND bo.deleted_at IS NULL
  SET boa.deleted_at = NOW(3),
      boa.updated_at = NOW(3)
  WHERE boa.form_view_id = {form_view_id}
    AND boa.deleted_at IS NULL
    AND NOT EXISTS (
      SELECT 1
      FROM t_business_object_attributes_temp boat
      JOIN t_business_object_temp bot
        ON boat.business_object_id = bot.id
       AND boat.form_view_id = bot.form_view_id
       AND boat.version = bot.version
       AND bot.deleted_at IS NULL
      WHERE boat.form_view_id = boa.form_view_id
        AND boat.version = {current_version}
        AND boat.deleted_at IS NULL
        AND boat.form_view_field_id = boa.form_view_field_id
        AND bot.object_name = bo.object_name
    );
  ```

   3.6 **更新理解状态**

   ```sql
   UPDATE t_data_view_understanding
   SET understand_status = 3,
       updated_at = NOW(3)
   WHERE view_id = {view_id}
     AND deleted_at IS NULL;
   ```

   **事务提交**

   **异常处理**：如果任何步骤失败，回滚所有已执行的操作
4. **返回提交成功响应**

**响应数据**：

```json
{
  "code": 200,                                 // int - 响应状态码
  "message": "提交成功",                        // string - 响应消息
  "data": {
    "understand_status": 3                    // int - 理解状态（0-未理解,1-理解中,2-待确认,3-已完成,4-已发布）
```

#### 4.3.4 删除业务对象识别结果

**接口路径**：`DELETE /api/data-semantic/v1/{id}/business-objects`

**并发控制**：同一用户对同一库表1秒内只允许点击一次，使用 Redis 限流，Key 格式为 `库表UUID:用户ID`。

**处理逻辑**：

1. **校验当前 understand_status**

   ```sql
   SELECT understand_status FROM t_data_view_understanding WHERE view_id = {view_id} AND deleted_at IS NULL;
   ```

   - `0-未理解`：没有数据可删除
   - `1-理解中`：正在生成数据，不允许删除
   - `2-待确认`：允许删除
   - `3-已完成`：已完成理解，不允许删除
   - `4-已发布`：已发布，不允许删除
2. **检查是否存在当前使用的业务对象临时数据**

   ```sql
   SELECT COUNT(*) AS count
   FROM t_business_object_temp
   WHERE form_view_id = {form_view_id} AND in_use = 1 AND deleted_at IS NULL;
   ```

   如果 count = 0，提示"没有可删除的数据"
3. **使用数据库事务执行删除操作**

   **事务开始：**

   3.1 **逻辑删除当前使用的业务对象临时数据**

   ```sql
   UPDATE t_business_object_temp
   SET deleted_at = NOW(3)
   WHERE form_view_id = {form_view_id}
     AND in_use = 1
     AND deleted_at IS NULL;
   ```

   3.2 **逻辑删除当前使用的业务对象属性临时数据**

   ```sql
   UPDATE t_business_object_attributes_temp
   SET deleted_at = NOW(3)
   WHERE form_view_id = {form_view_id}
     AND in_use = 1
     AND deleted_at IS NULL;
   ```

   3.3 **检查正式表是否有业务对象数据，决定最终状态**

   ```sql
   -- 检查正式表是否有业务对象
   SELECT COUNT(*) AS object_count
   FROM t_business_object
   WHERE form_view_id = {form_view_id} AND deleted_at IS NULL;
   ```

   - 如果 `object_count > 0`：保持 `3-已完成`
   - 如果 `object_count = 0`：更新为 `0-未理解`

   ```sql
   -- 根据是否有正式数据更新状态
   UPDATE t_data_view_understanding
   SET understand_status = CASE
     WHEN (SELECT COUNT(*) FROM t_business_object WHERE form_view_id = {form_view_id} AND deleted_at IS NULL) > 0
       THEN 3  -- 保持已完成
     ELSE 0    -- 回退到未理解
   END
   WHERE view_id = {view_id} AND deleted_at IS NULL;
   ```

   **事务提交**
4. **说明**：

   - `t_form_view_info_temp` 和 `t_form_view_field_info_temp` 数据**保留**，不删除
   - 删除操作使用 `in_use = 1` 条件，确保只删除当前使用的识别结果
   - 历史版本数据（`in_use = 0`）保留，不参与当前业务流程
   - 删除后根据正式表是否有数据决定最终状态：
     - 有正式数据 → `3-已完成`
     - 无正式数据 → `0-未理解`
   - 如果需要重新生成，需要调用接口6重新生成数据

   ```java
   @KafkaListener(topics = "data-understanding-responses", groupId = "data-understanding-consumer-group")
   public void handleAIResponse(String message) {
       // 1. 解析 Kafka 消息
       AIResponse response = JSON.parseObject(message, AIResponse.class);

       // 2. 校验 message_id 是否存在且未处理（防止重复消费）
       if (isMessageProcessed(response.getMessageId())) {
           log.warn("消息已处理，跳过: {}", response.getMessageId());
           return;
       }

       // 3. 校验库表当前理解状态（必须是 1-理解中 才处理）
       int currentStatus = getUnderstandStatus(response.getFormViewId());
       if (currentStatus != 1) {
           log.warn("库表理解状态不是1-理解中，跳过处理: form_view_id={}, current_status={}",
               response.getFormViewId(), currentStatus);
           // 标记消息已处理（不处理）
           markMessageAsProcessed(response.getMessageId(), 1);
           return;
       }

       // 4. 根据 status 字段判断成功或失败
       if ("success".equals(response.getStatus())) {
           // 4.1 成功：保存数据到临时表，更新状态为 2-待确认
           try {
               transactionTemplate.execute(status -> {
                   saveTableSemantic(response.getFormViewId(), response.getData().getTableSemantic());
                   saveFieldsSemantic(response.getFormViewId(), response.getData().getFieldsSemantic());
                   saveNoPatternFields(response.getFormViewId(), response.getData().getNoPatternFields());
                   saveBusinessObjects(response.getFormViewId(), response.getData().getBusinessObjects());
                   return null;
               });

               // 更新 understand_status 为 2-待确认
               updateUnderstandStatus(response.getFormViewId(), 2);

               // 标记消息已处理
               markMessageAsProcessed(response.getMessageId(), 1);

           } catch (Exception e) {
               log.error("处理 AI 响应失败: {}", response.getMessageId(), e);
               // 标记消息处理失败
               markMessageAsProcessed(response.getMessageId(), 2);
               // understand_status 保持为 1-理解中
           }
       } else {
           // 4.2 失败：记录错误日志，更新状态为 5-理解失败
           log.error("AI 处理失败: message_id={}, error={}",
               response.getMessageId(), response.getError().getMessage());

           // 更新 understand_status 为 5-理解失败
           updateUnderstandStatus(response.getFormViewId(), 5);

           // 标记消息已处理（失败）
           markMessageAsProcessed(response.getMessageId(), 2, response.getError().getMessage());
       }
   }
   ```

   ```sql
   UPDATE t_business_object_attributes_temp
   SET
     business_object_id = {target_object_id},
     user_id = {user_id},
     updated_at = NOW(3)
   WHERE id = {attribute_id}
     AND form_view_id = {form_view_id}
     AND deleted_at IS NULL;
   ```
6. **检查原业务对象是否为空，如果为空可以选择性处理**

   ```sql
   SELECT COUNT(*) AS attr_count
   FROM t_business_object_attributes_temp
   WHERE business_object_id = {old_object_id}
     AND form_view_id = {form_view_id}
     AND version = {current_version}
     AND deleted_at IS NULL;
   ```

   **说明**：

   - 如果原业务对象的属性数量为 0，允许业务对象为空（不强制删除）
   - 前端可以选择性隐藏空业务对象，或保留显示
7. **说明**：

   - 不递增版本号
   - `user_id` 不为空，表示用户手动操作
   - 允许业务对象变为空（将所有属性移走后，原业务对象可以为空）
   - 原业务对象即使为空也不自动删除，保留记录

**响应数据**：

#### 4.3.11 批量业务对象匹配

**接口路径**：`POST /api/data-semantic/v1/batch-object-match`

**接口说明**：批量匹配业务对象，输入为业务对象名称或视图ID列表。有视图ID时直接返回，有对象名字时优先从业务对象表模糊匹配，未匹配时从视图表模糊匹配并返回。

**请求参数**：

| 参数名   | 类型   | 必填 | 说明                       |
| -------- | ------ | ---- | -------------------------- |
| entries  | array  | 是   | 业务对象列表，最多100条    |
| entries[].name         | string | 是   | 业务对象名称（非空字符串） |
| entries[].data_source  | object | 否   | 给定的视图数据             |
| entries[].data_source.id   | string | 否   | 视图ID（对应 t_data_view.f_view_id） |
| entries[].data_source.name | string | 否   | 视图名称                   |

**请求示例**：

```json
{
  "entries": [
    {
      "name": "客户信息",
      "data_source": {
        "id": "xxx",
        "name": "客户主档视图"
      }
    },
    {
      "name": "订单管理"
    }
  ]
}
```

**响应数据**：

```json
{
  "code": 200,
  "data": {
    "entries": [
      {
        "name": "客户信息",
        "data_source": [
          {
            "id": "mdl_xxx",
            "name": "客户主档视图",
            "object_name": "客户信息"
          }
        ]
      },
      {
        "name": "订单管理",
        "data_source": [
          {
            "id": "mdl_yyy",
            "name": "订单视图",
            "object_name": "订单管理"
          }
        ]
      }
    ],
    "need_understand": ["view_id_1", "view_id_2"]
  }
}
```

**响应字段说明**：

| 字段名 | 类型 | 说明 |
| ------ | ---- | ---- |
| entries | array | 匹配结果列表 |
| entries[].name | string | 原始输入名称 |
| entries[].data_source | array | 匹配的视图列表 |
| entries[].data_source[].id | string | 视图ID（mdl_id） |
| entries[].data_source[].name | string | 视图名称（TechnicalName） |
| entries[].data_source[].object_name | string | 业务对象名称 |
| need_understand | array | 需要理解的视图ID列表（已去重） |

**处理逻辑**：

1. **输入验证**：过滤空name的条目
2. **Step 1 - data_source有值**：如果给定了 `data_source.id`，直接追加到结果（返回id、name、object_name）
3. **Step 2 - 业务对象表模糊匹配**：
   ```sql
   SELECT id, object_name, form_view_id
   FROM t_business_object
   WHERE object_name LIKE '%{name}%' AND deleted_at IS NULL;
   ```
   - 找到匹配时，通过 `form_view_id` 查询视图信息获取 `mdl_id` 和 `TechnicalName`
4. **Step 3 - 视图表模糊匹配**：
   ```sql
   SELECT
     dv.f_view_id AS id,
     dvu.understand_status,
     dv.f_technical_name AS technical_name,
     dvu.table_business_name AS business_name
   FROM t_data_view dv
   LEFT JOIN t_data_view_understanding dvu ON dvu.view_id = dv.f_view_id AND dvu.deleted_at IS NULL
   WHERE dvu.table_business_name LIKE '%{name}%'
     AND dv.f_delete_time = 0;
   ```
5. **Step 4 - 状态检查**：
   - 无论什么状态都追加视图到 `data_source`
   - 仅当 `understand_status != 3` 时记录视图ID到 `need_understand`
   - 如果 `understand_status == 3`（已理解），查询业务对象表获取 `object_name`

**匹配优先级**：
1. 有 `data_source.id` → 直接返回
2. 业务对象表模糊匹配 → 返回匹配结果
3. 视图表模糊匹配 → 返回视图（未理解时记录到need_understand）

**错误响应**：

| HTTP状态码 | 错误码 | 说明 |
| ---------- | ------ | ---- |
| 400 | - | 请求参数错误（如entries为空数组） |

### 4.4 版本号管理规则

**版本号设计原则**：采用统一版本号管理，所有临时表共享同一个版本号。

| 接口                         | 版本号变化 | 说明                                                                                                                                                                  |
| ---------------------------- | ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 接口6 - 一键生成理解数据     | 递增       | 生成新版本时，递增所有临时表（`t_form_view_info_temp`、`t_form_view_field_info_temp`、`t_business_object_temp`、`t_business_object_attributes_temp`）的版本号 |
| 接口5 - 重新识别业务对象     | 递增       | 重新识别业务对象时，递增 `t_business_object_temp` 和 `t_business_object_attributes_temp` 的版本号                                                                 |
| 接口7 - 保存库表信息补全数据 | 不递增     | 在当前最新版本上进行更新                                                                                                                                              |
| 接口8 - 保存业务对象及属性   | 不递增     | 在当前最新版本上进行更新                                                                                                                                              |
| 接口9 - 调整属性归属业务对象 | 不递增     | 在当前最新版本上进行更新                                                                                                                                              |

**总结**：

- 接口6（一键生成）：递增所有临时表的版本号
- 接口5（重新识别）：只递增业务对象相关临时表的版本号
- 其他编辑操作：在当前最新版本上进行更新，不递增版本号

---

### 4.5 临时表数据处理策略

**临时表数据保留策略**：所有临时表数据在提交后**保留**，不删除。

**重新生成时的处理方式**：

| 场景                     | 涉及临时表         | 处理方式                         |
| ------------------------ | ------------------ | -------------------------------- |
| 接口6 - 一键生成理解数据 | 所有临时表         | 新增记录，版本号递增，旧记录保留 |
| 接口5 - 重新识别业务对象 | 业务对象相关临时表 | 新增记录，版本号递增，旧记录保留 |

**查询时数据来源**：

- 查询接口总是返回**最新版本**的数据
- 旧版本数据作为历史记录保留，不返回给前端

**说明**：

- 采用"新增记录"而非"更新记录"的方式，可以保留完整的修改历史
- 每次重新生成都会创建新版本的记录，便于追溯和对比
- 查询时通过 `in_use = 1` 筛选当前使用的版本，历史版本数据保留但不返回给前端

### 4.7 理解状态流转

> 理解状态统一存储于 `t_data_view_understanding.understand_status`，按 `view_id` 唯一维护。

| 当前状态 | 可调用接口 | 调用后状态                                   |
| -------- | ---------- | -------------------------------------------- |
| 0-未理解 | 接口6      | 1-理解中（异步AI分析）                       |
| 1-理解中 | -          | 2-待确认（AI成功后自动流转）                 |
| 1-理解中 | -          | 5-理解失败（AI失败后自动流转）                |
| 2-待确认 | 接口3      | 3-已完成                                     |
| 2-待确认 | 接口5      | 2-待确认（数据刷新）                         |
| 2-待确认 | 接口4      | 0-未理解或3-已完成（取决于正式表是否有数据） |
| 3-已完成 | 接口4      | 0-未理解或3-已完成（取决于正式表是否有数据） |
| 3-已完成 | -          | 4-已发布（业务对象建模阶段）                 |
| 4-已发布 | -          | -                                            |
| 5-理解失败 | 接口6      | 1-理解中（重新尝试）                         |

**说明**：

- `0-未理解` → `1-理解中`：用户点击"开始理解"按钮后立即设置
- `1-理解中` → `2-待确认`：AI异步处理成功后自动设置，前端需要轮询或接收通知来获取状态变化
- `1-理解中` → `5-理解失败`：AI异步处理失败后自动设置，用户可重新尝试
- `5-理解失败` → `1-理解中`：用户点击"开始理解"按钮后重新尝试
- **接口4删除回退逻辑**：删除业务对象临时数据后，检查正式表是否存在业务对象数据
  - 有正式数据 → 保持 `3-已完成`
  - 无正式数据 → 回退到 `0-未理解`

### 4.8 正式表增量更新策略

**设计背景**：保持原有业务流程不变，提交阶段统一将临时表数据同步到正式表。

**核心机制**：按业务唯一键执行 Upsert（存在即更新，不存在即新增）。

#### 4.8.1 Upsert 唯一键说明

| 正式表 | 唯一键 |
| --- | --- |
| `t_data_view_understanding` | `view_id` |
| `t_data_view_field_semantic` | `(view_id, view_field_id)` |
| `t_business_object` | `(form_view_id, object_name)`（结合业务约束） |
| `t_business_object_attributes` | `(business_object_id, form_view_field_id)`（结合业务约束） |

#### 4.8.2 提交流程

1. 读取四类临时表当前版本数据：`t_form_view_info_temp`、`t_form_view_field_info_temp`、`t_business_object_temp`、`t_business_object_attributes_temp`  
2. 依次同步到四类正式表：`t_data_view_understanding`、`t_data_view_field_semantic`、`t_business_object`、`t_business_object_attributes`  
3. 对每条记录执行 Upsert：存在则更新，不存在则新增  
4. 同步完成后更新 `t_data_view_understanding.understand_status = 3`

#### 4.8.3 示例场景

**场景：用户首次提交后再次编辑提交**

| 步骤 | 临时表状态 | 正式表操作 | 说明 |
|------|-----------|-----------|------|
| 1 | 首次提交数据 | INSERT | 正式表无记录，新增 |
| 2 | 二次提交，部分数据有变化 | UPDATE | 命中唯一键，更新变化字段 |
| 3 | 二次提交，新增对象/属性 | INSERT | 未命中唯一键，新增 |

#### 4.8.4 优势总结

| 对比项 | 全量替换（旧方案） | Upsert 增量同步（新方案） |
|--------|-------------------|--------------------------|
| 业务流程 | 需额外删除/重建 | 保持原流程不变 |
| 变更范围 | 可能影响全量数据 | 仅更新变化项 |
| 可维护性 | 规则复杂 | 规则统一清晰 |
