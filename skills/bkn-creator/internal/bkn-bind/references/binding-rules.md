# 对象级绑定判定规则

## 绑定状态

| 状态 | 条件 |
|------|------|
| `bound` | 唯一主绑定视图 + 稳定 view_id + 语义一致 + 字段兼容 |
| `pending` | 多候选无法裁决 / 需语义补齐 / 仅中低置信度 |
| `rejected` | 无可接受候选 / 业务边界冲突 / 上下文不足 |

## 差异动作

| 动作 | 适用场景 |
|------|----------|
| `keep` | 对象与视图语义一致，边界不变 |
| `add` | 视图中发现新业务实体，草案未覆盖 |
| `rename` | 对象本质正确但命名偏差 |
| `merge` | 多个草案对象实质指向同一实体 |
| `drop_candidate` | 候选为噪音/重复/技术派生 |

## 候选视图筛选

**优先保留：** 语义相关、覆盖主键和核心属性、命名稳定、非技术中间层
**优先排除：** 纯技术字段、命名模糊、只覆盖边缘属性、属于其他业务域

## 置信度

- **高**：语义一致 + 主键存在 + 核心属性覆盖 + 无冲突
- **中**：多候选接近 / 主键不完整 / 需 data-semantic 补齐
- **低**：仅凭名称猜测 / 无法确认主键

低置信度不得直接回填。

## 多视图冲突

- 保留并列候选，不擅自压成唯一
- 比较主键稳定性、核心属性覆盖率、语义清晰度
- 无法裁决 → pending

## 禁止

- 不仅凭名称相似度确认绑定
- 不用视图名称替代 view_id
- 不因下游需回填而伪造绑定成功
